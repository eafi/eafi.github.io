异常处理机制/c++:throw-exceptions

SuperEafi
2020-08-07
Report
C++

C++的异常处理机制是非常有用的工具，但是受限于之前不能深入理解其中的机理，在实际代码书写中还是会使用更佳简单的return error code的做法。为了比较深入浅出的方便自己在未来的代码工作中更合理和自信的使用异常处理机制，多方总结C++异常处理的经验并汇总为此文。

本文的组织方式为：

  • RAII带来的好处
  • 为什么我们需要异常处理机制
  • 异常处理的性能损耗
  • 怎么抛和捕获异常
  • 构造函数和析构函数使用异常处理

为何使用异常处理？

简而言之，异常处理可以简化代码。过去的设计中，一种常见的异常处理设计是使用if语句进行判断，然后对错误信息进行return error code。比如，一种return error code是这么设计的：

这会导致至少以下问题：

  1. 如果f1不能够处理错误（即错误不能够在f1的if语句中进行解决），那么它必须在他的返回值中继续返向传播错误，这意味着如果函数调用非常深时，if和return error code将弄的代码非常难看

  2. return error code的做法占用了返回值，使得想要返回修改值只能通过指针或引用传递，返回值只能作为异常处理使用。当然，可以在意义明确的情况下指明特殊的取值范围为异常，比如我们把返回值封装成一个对象，使其能够包含更多信息。但这明显增加了复杂度。

异常处理的性能损耗

讨论异常处理的性能损耗，是因为常见的error-return看上去轻量简单。但事实真的如此吗？结论是，对于现代C++来说，异常处理的性能损耗是非常小的。对于完全没有异常的正常执行的代码，几乎无损耗可言，在这一点上就完胜于error-return以及与其匹配的测试代码了。异常处理的损耗出现在引发了异常。因此，合理的使用异常处理并不会引起严重的性能下降，相反避免繁重的error-return和安全测试代码能够提升性能。

有三种方式可以捕获(catch)throw抛出的对象，分别是传值、传引用和传指针来捕获异常对象。但这里有额外的说明。

最好不要catch指针

最好不要catch指针，因为无法得知是否需要在catch中做其他类似释放或新建这个指针的操作。增加的工作量是无意义的。比如，如果传入的指针是没有实际对象，需要catch自己创建，那么问题更严重了，如果程序异常的原因就是堆空间不足，那么在catch中new无疑是严重的问题。总之，使用catch *e一定要万般小心传入对象的状态、位置。

另外，catch不接受rvalue-reference，即类似catch(Window &&e)无法通过编译。

因此，可以总结的是，通常情况catch使用引用捕获异常适用于大部分情况。同时，我们要密切注意throw抛出的对象更像是拷贝从而没有多态性，如果要使用多态性要封装throw到虚函数，而不是使用抛出指针的危险行为。

其他注意事项

构造函数抛出异常

对于不能够在构造函数中正确初始化的对象确实是应该使用异常来处理，因为构造函数中不提供返回值。除了使用异常来处理外，其实还可以让错误初始化的对象呈现“僵尸状态”，这时所有试图构造这个对象的地方都必须要使用if来检查是否是僵尸状态，这显然又增加了调用者的负担。

在构造函数中抛出异常，不会调用这个对象的析构函数，原因显而易见：可能存在一些资源还没有正确构造，贸然调用对象析构函数可能会有问题。已经分配的资源会自动释放，原因是：每一个对象内部在构造所需资源时，那些资源因为按照RAII设计，而自我进行释放(自我析构函数的调用)。设计对象需要智慧，比如如果我们在对象中使用了指针，显然指针这个东西并不符合RAII：

小结

本文梳理了C++中使用的异常处理机制，尽量讲清楚其中的原委。需要注意的是并不是任何时候异常处理机制都是灵丹妙药，异常处理总归是发生异常时的一种解决机制，他不能用来做代码使用错误的判断机制（代码逻辑错误、使用错误应该直接assert掉）。异常应该是指资源分配时出现了问题，又或者是强制类型转换出错、出界、溢出等错误。另外，异常处理可以用来传播异常，以便其他能够处理这个异常的函数进行调用，但这种传播不需要靠返回值实现，我们只需要throw或者rethrow即可.

一定要谨慎使用异常处理，只有认为:这里可能会发生异常，我需要捕获它，并且我的捕获部分能够正确处理和继续保持程序的执行，我的捕获部分无需再广播异常（意味着捕获必须确保能够handle异常），并且我不想使用return-error-code来处理这个异常

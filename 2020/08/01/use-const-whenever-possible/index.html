摘要

本文为Effective C++中Use const whenever possible的笔记。主要用以记录书中细节和一些思考。

为什么需要const？

const关键字最基本的理解就是常量，即我们不想让后面的操作有意或无意地修改const修饰的内容.const的作用范围非常广,const可被施加在任何作用域内的对象、函数参数、函数返回类型、成员函数本体。

const是一个语义约束，编译器会强制实施该约束。这是const非常棒的特性，因为我们可以使用const实现控制哪些值是不能改变的，任何试图破坏这种约束的错误用法都会被编译器侦测。因此，要让这些对象受到编译器的保护，且这种保护的能够为我们所用，需要理清楚在各种情况下const修饰的特点。

const值

const能够修饰一般的变量从而提升为常量值。被const修饰的值可以是指针、一般的变量、函数的参数或者是函数的返回类型，下面总结一些重要的细节点。

bitwise const 和 logical constness

const成员函数有两种解释const的逻辑，一个被称为bitwise const，这个逻辑认为const成员函数不得更改对象任何成员变量，一旦更改就非法。bitwise const非常适合编译器检测的实现，一旦编译器监测到const成员函数出现了non-static成员变量的赋值操作，那么立刻报错。

旁注，bitwise const允许const成员函数中修改非const的static成员

这种检测方法存在漏洞：一个更改了“指针所指之物”的成员函数虽然不能算const，但如果只有指针（而非所指之物）隶属于对象，那么称此函数为bitwise const不会引发编译器异议。这意味着，如果一个成员变量是指针，那么bitwise基于成员变量是否赋值的检测就会有漏洞，因为此时我们可以不改成员指针（即所指物地址），而直接修改所指物的内容（改所指物的内容不会改变成员变量指针，因此编译器通过bitwise const检测）:

logical constness认为const成员函数可以修改他所处理的对象内的某些数据，但只有在客户端侦测不出来的情况下才可以。这意味着const成员函数可以改成员数据，但从调用这个const成员函数的用户角度这个改变是不可见的。比如，接着上面的class CTextBlock，我们希望有一个返回长度的const成员函数（返回指针所指字符串长度的成员函数显然应该是const的，因为它从用户角度而言并不期望修改任何成员变量，而只期望得到这个字符串的长度信息）。

为了实现高速缓存，我们不希望每次调用长度成员函数都要重新计算一次长度，如果长度已经计算过，那么直接返回它的保存至即可。因此这个length()需要满足两个需求：

  1. 需要是const成员函数，因为仅仅返回长度的成员函数显然不会修改其他成员变量；

  2. 需要在这个成员函数中引入控制变量，因为为了满足高速缓冲需求

在const和non-const成员函数中避免重复

现在我们知道了为了实现自建对象是const或non-const需要对应的有const和non-const成员函数共同存在。但这也带来了全新的问题，那就是如果利用const常量性来重载的两个函数很长，那么将导致过分的代码重复，从而带来维护性上的困难。举个例子，如果我们需要对上述class TextBlock在调用operator[]时进行辩解检查、记录访问信息、数据完整性检查等等安全检查(这些检查是理所因当的，因为我们需要保证operator[]的安全性):

这种代码重复不是我们所期望的，一种解决办法是将这些检查、记录操作移动到另一个私有成员函数，然后让这两个operator[]调用它。但还是不好，因为首先，调用过程还是重复了代码（一次函数调用和两次return语句）；其次，我们自己理清可以发现这两个operator[]除了返回值不同外，其实非常的相似，我们希望可以通过其中一个调用另一个。

假如我们想要实现让const版本调用non-const版本以避免重复，这会招致风险：const对象的初衷是对象不应该被修改，但想要调用non-const版本，这意味着我们需要把const对象的常量性给移除，然后才能合法的交由non-const版本。问题来了:我们不能保证non-const版本中的操作不会修改对象成员，这相当于是铤而走险的放弃掉const优良的常量性保证，最后返回得到的const对象还是我们原来的那个吗？不知道。

一种合理的实现是令non-const版本调用其const兄弟。这个逻辑安全的，因为non-const成员本身不限制能否被修改，那么它被转换成const并传入到一个不能被修改的const成员函数无伤大雅。

上述代码安全地解决的代码重复性的问题，其中在non-const实现中实现了两个强制类型转换:第一次是使用static_cast<>强制转换成const对象，因为只有const对象才有资格调用const成员函数；第二次使用const_cast<>移除刚才加上的const常量性，这一步的移除是安全的，因为我们能够保证前后对象恢复到了原始状态。

小结

  1. 声明为const可帮助编译器侦测出错误用法。而且const非常强大，可以被施加于任何作用域内的对象、函数参数、返回类型和成员函数本体上；

  2. 编译器是施行bitwise constness的，但作为代码作者，应该从logical constness进行考虑

  3. const和non-const版本会导致代码重复，上文实现了一种安全有效的避免代码重复的方法
